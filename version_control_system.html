Version control system

A version control system is a system that records changes overtime so that you can recall specific versions later.

For example if you are a web designer, and find yourself in any of these situations:
  • You like a particular design layout. And you don’t want future changes to mess with your current design
  • You go with a design pattern and find down the road that, it may be flawed, In situations like these, It is smart to use a version control system as they allow you to revert the selected files back to previous state(or even the entire project), compare changes overtime, or even see where the error was introduced.

It also finds use for people who collaborate towards an open source project from any part of the world as it allows them to make changes to software and merge them together in the end, even if they are worlds apart. GIT here, allows you to see who last modified something that might be causing an issue and when.
In short, Using a VCS also generally means that if you screw things up or lose files, you can easily recover.

  • Centralized VCS ,eg: SVN
  • Distributed VCS ,eg: GIT
  • Local vcs

Central VCS
Centralized version control system (CVCS) uses a central server to store all files and enables team collaboration. It works on a single repository to which users can directly access a central server. Here, every programmer can extract and update their workstations with the data present in the repository or can make changes to the data. Every operation is performed directly on the repository. It has some major drawbacks. Some of them are:
  • You always need network connection to perform any action or changes, because it is not locally available.
  • Due to its centralization, if the central server is crashed or corrupted, The entire data in the project will be lost.

Distributed version control system
These systems do not necessarily rely on a central server to store all the versions of a project file. In Distributed VCS, every contributor has a local copy or “clone” of the main repository i.e. everyone maintains a local repository of their own which contains all the files and metadata present in the main repository.

As you can see in the above diagram, every programmer maintains a local repository on its own, which is actually the copy or clone of the central repository on their hard drive. They can commit and update their local repository without any interference.
They can update their local repositories with new data from the central server by an operation called “pull” and affect changes to the main repository by an operation called “push” from their local repository.

The act of cloning an entire repository into your workstation to get a local repository gives you the following advantages:

  • All operations (except push & pull) are very fast because the tool only needs to access the hard drive, not a remote server. Hence, you do not always need an internet connection.
  • Committing new change-sets can be done locally without manipulating the data on the main repository. Once you have a group of change-sets ready, you can push them all at once.
  • Since every contributor has a full copy of the project repository, they can share changes with one another if they want to get some feedback before affecting changes in the main repository.
  • If the central server gets crashed at any point of time, the lost data can be easily recovered from any one of the contributor’s local repositories.

Local Version Control Systems
Many people’s version-control method (including mine ) of choice is to copy files into another directory . This approach is very common and simple. But has its disadvantages:
  • It is easy to forget which directory you’re in and accidentally write to the wrong file or copy over files you don’t mean to (I’ve been there) .
  • If your system crashes, and you have no back up, your entire project is gone.
  • It only allows a single user to collaborate
